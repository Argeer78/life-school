# Purpose

Provider Failure Handling defines how Life School responds when an external language model cannot successfully fulfill a generation request.

Its purpose is to isolate provider failures from constitutional behavior.

A provider failure is an infrastructure event.

It is not a constitutional event.

---

# Principle

Steward shall remain constitutionally consistent regardless of provider failures.

No provider failure shall become learner-visible unless required for truthful communication.

---

# Scope

Provider failures include:

- network failure,
- timeout,
- provider unavailable,
- rate limiting,
- malformed output,
- schema validation failure,
- unsupported model version,
- authentication failure,
- provider refusal,
- interrupted streaming.

---

# Failure Categories

Provider failures shall be classified.

## PB-FAIL-001

Connection Failure

Unable to contact provider.

---

## PB-FAIL-002

Timeout

Generation exceeded permitted limits.

---

## PB-FAIL-003

Invalid Response

Provider returned invalid structured output.

---

## PB-FAIL-004

Schema Violation

Output failed PB-002 validation.

---

## PB-FAIL-005

Provider Refusal

Provider declined to generate.

---

## PB-FAIL-006

Internal Provider Error

Provider reported an internal failure.

---

## PB-FAIL-007

Unknown Failure

Failure cannot be classified.

---

# Failure Procedure

When a provider failure occurs:

1. classify failure;

2. record privileged audit information;

3. determine whether retry is permitted;

4. if recovery fails,
   invoke EN-005.

The learner shall never receive malformed provider output.

---

# Retry Policy

Retries shall be limited.

Recommended default:

- one immediate retry for transient failures,
- no retries for schema violations,
- no retries for malformed output.

Retry policy shall remain deterministic.

---

# Logging

Failure logs shall include:

- timestamp,
- provider,
- model,
- failure category,
- contract version,
- request identifier.

Logs shall remain privileged.

---

# Learner Communication

Infrastructure failures shall be communicated honestly.

Example:

"I'm unable to generate a reliable response at the moment. Rather than guessing, I'd prefer to be honest about that limitation."

The learner shall not receive:

- stack traces,
- API errors,
- internal identifiers,
- provider diagnostics.

---

# Independence

Failure handling shall remain independent of:

- provider vendor,
- API format,
- deployment platform,
- operating system.

---

# Success Criteria

The Provider Failure Handler succeeds when:

- provider failures remain isolated,
- constitutional behavior continues,
- privileged information remains protected,
- learners receive respectful communication.

---

# Failure Conditions

The Provider Failure Handler fails if:

- malformed provider output reaches learners,
- provider failures alter constitutional behavior,
- internal implementation details become visible,
- retry behavior becomes nondeterministic.

---

# Constitutional Guarantee

Provider failures shall never become constitutional failures.

When recovery is not possible, the constitutional pipeline shall continue through EN-005.
