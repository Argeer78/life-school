# Purpose

Provider Response Validation verifies that every response produced by an external language model satisfies the Provider Contract before entering the constitutional pipeline.

Its purpose is to reject malformed, incomplete, or unauthorized provider outputs before they influence Steward.

Validation protects the constitutional boundary.

Validation does not determine constitutional correctness.

---

# Principle

No provider response shall enter constitutional review unless it first satisfies the Provider Contract.

Validation occurs before constitutional review.

---

# Scope

Validation applies to:

- hosted language models,
- local language models,
- future providers,
- streamed responses,
- cached responses.

---

# Validation Inputs

The validator receives:

- provider output,
- Provider Contract version,
- expected schema,
- generation metadata.

---

# Validation Outputs

Validation returns one of:

PASS

The provider response satisfies the contract.

---

FAIL

The provider response violates the contract.

The response shall not continue through the pipeline.

---

# Required Validation

Validation shall verify:

- required fields exist,
- field types are correct,
- JSON is valid,
- schema version matches,
- prohibited fields are absent,
- response size limits,
- UTF-8 validity,
- serialization correctness.

---

# Forbidden Provider Content

Validation shall reject outputs containing:

- chain of thought,
- internal reasoning,
- privileged audit information,
- engine implementation details,
- constitutional authority,
- unsupported metadata,
- malformed schema.

---

# Optional Validation

Implementations may additionally verify:

- maximum token count,
- language constraints,
- formatting constraints,
- citation structure,
- markdown safety.

These checks shall not replace required validation.

---

# Failure Handling

Validation failure shall:

- reject the provider response,
- record privileged audit information,
- invoke EN-005 if recovery is impossible.

The learner shall never receive invalid provider output.

---

# Determinism

Validation shall be deterministic.

Identical provider outputs shall always produce identical validation results.

---

# Version Compatibility

Validation shall enforce the declared Provider Contract version.

Unsupported versions shall fail validation.

---

# Independence

Validation shall remain independent of:

- provider vendor,
- provider model,
- deployment environment,
- API transport.

Only the Provider Contract is authoritative.

---

# Success Criteria

Validation succeeds when:

- malformed responses are rejected,
- valid responses proceed,
- provider-specific behavior remains isolated,
- constitutional boundaries remain intact.

---

# Failure Conditions

Validation fails if:

- invalid responses reach constitutional review,
- provider-specific assumptions bypass validation,
- schema violations are ignored,
- privileged data becomes learner-visible.

---

# Constitutional Guarantee

Every provider response shall be validated before constitutional review.

Only validated provider outputs may enter the constitutional pipeline.
