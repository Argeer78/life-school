# Purpose

Structured Prompt Construction defines how Life School constructs provider requests from constitutional engine outputs.

Its purpose is to ensure that every provider receives identical constitutional guidance regardless of provider implementation.

Prompt construction prepares context.

It does not determine constitutional behavior.

---

# Principle

The constitutional engine determines behavior.

The prompt communicates behavior.

Prompt construction shall never redefine constitutional intent.

---

# Input

Prompt construction receives:

- learner message,
- current conversation,
- selected Conversation Strategies,
- Behavior Plan,
- constitutional constraints,
- provider contract version,
- output schema.

No additional constitutional authority may be introduced.

---

# Prompt Sections

Every provider request shall contain the following sections.

## Identity

Defines Steward.

Derived from:

- ST-001

---

## Constitutional Context

Summarizes applicable constitutional constraints.

Derived from:

- selected Conversation Strategies,
- Behavior Plan.

---

## Learner Context

Contains only the current conversation.

No historical memory shall be included unless explicitly supported by constitutional policy.

---

## Behavioral Instructions

Defines required behaviors.

Examples:

- preserve learner dignity,
- preserve learner freedom,
- preserve learner responsibility,
- acknowledge uncertainty,
- avoid manipulation,
- avoid invented authority.

These instructions originate exclusively from EN-002.

---

## Output Contract

Defines the PB-001 response schema.

No free-form provider output shall be requested.

---

# Forbidden Prompt Content

Prompt construction shall not include:

- chain of thought requests,
- hidden reasoning requests,
- internal audit information,
- review results,
- revision history,
- provider-specific optimization,
- implementation secrets,
- privileged inspection data.

---

# Prompt Determinism

Equivalent inputs shall produce equivalent prompts.

Minor implementation differences shall not alter constitutional meaning.

---

# Prompt Size

Prompt construction should include only information necessary to fulfill constitutional behavior.

Unnecessary context shall be omitted.

---

# Prompt Versioning

Prompt templates shall be versioned.

Breaking changes require a new prompt version.

Prompt versions shall remain independent of provider versions.

---

# Provider Independence

Prompt construction shall not depend upon:

- GPT-specific syntax,
- Claude-specific syntax,
- Gemini-specific syntax,
- vendor-specific prompting techniques.

The constitutional message shall remain identical.

---

# Validation

Every constructed prompt shall be validated before transmission.

Validation shall confirm:

- required sections,
- prohibited sections absent,
- schema version,
- deterministic construction,
- constitutional completeness.

---

# Success Criteria

Prompt construction succeeds when:

- identical constitutional inputs produce equivalent prompts,
- providers receive sufficient context,
- no constitutional authority crosses the Provider Boundary,
- prompts remain provider-independent.

---

# Failure Conditions

Prompt construction fails if:

- provider-specific assumptions appear,
- constitutional behavior changes because of prompt wording,
- privileged information enters the prompt,
- prompt construction bypasses EN-001 or EN-002.

---

# Constitutional Guarantee

Every provider prompt shall faithfully represent the constitutional decisions already made by the Engine.

Prompt construction communicates constitutional intent.

It never creates constitutional intent.
