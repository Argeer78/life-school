# Purpose

The Structured Provider Contract defines the complete interface between the constitutional engine and any external language model.

Its purpose is to ensure that every provider receives the same constitutional instructions and produces the same class of structured outputs.

The contract belongs to Life School.

Providers conform to the contract.

---

# Principle

The provider receives constitutional context.

The provider returns language.

The provider does not determine constitutional behavior.

---

# Inputs

The provider may receive only:

- learner message,
- current conversation,
- selected Conversation Strategies,
- Behavior Plan,
- constitutional constraints,
- permitted response style,
- output schema.

The provider shall not receive:

- hidden constitutional authority,
- privileged audit data,
- internal engine state unrelated to generation,
- future pipeline decisions.

---

# Required Output

The provider shall return a structured object.

Example schema:

{
  "response": "...",
  "confidence": 0.87,
  "uncertainty": true,
  "refusal": false,
  "notes": []
}

The exact schema may evolve through versioning.

---

# Response

The response field shall contain only learner-facing language.

It shall not contain:

- internal reasoning,
- chain of thought,
- review results,
- implementation details,
- privileged information.

---

# Confidence

Confidence represents the provider's confidence in the linguistic expression.

It does not determine constitutional authority.

Confidence shall never override constitutional review.

---

# Uncertainty

The provider shall explicitly identify uncertainty whenever appropriate.

The engine determines how uncertainty is communicated.

---

# Refusal

The provider may indicate that it cannot produce a valid response.

Provider refusal does not terminate the constitutional pipeline.

The Engine shall determine the next constitutional action.

---

# Notes

Optional provider metadata.

Notes are privileged.

They shall never be exposed directly to learners.

---

# Output Validation

Every provider response shall be validated before constitutional review.

Validation shall confirm:

- required fields,
- schema correctness,
- type correctness,
- size limits,
- prohibited content.

Invalid responses shall be rejected.

---

# Invalid Provider Output

Examples include:

- malformed JSON,
- missing response,
- unexpected fields,
- prohibited fields,
- oversized responses,
- invalid types.

Invalid outputs shall not enter constitutional review.

---

# Provider Independence

The contract shall remain independent of:

- OpenAI,
- Anthropic,
- Google,
- Meta,
- local models,
- future providers.

Every provider shall conform to the same contract.

---

# Versioning

The contract shall be versioned.

Breaking changes require a new version.

Existing providers shall continue to support previous approved versions where practical.

---

# Success Criteria

The contract succeeds when:

- every provider receives equivalent constitutional inputs,
- every provider returns valid structured outputs,
- providers remain interchangeable,
- constitutional behavior remains stable.

---

# Failure Conditions

The contract fails if:

- providers return uncontrolled text,
- provider-specific assumptions enter the engine,
- schema validation is bypassed,
- provider output becomes constitutional authority.

---

# Constitutional Guarantee

Every provider shall communicate through a validated structured contract.

Constitutional authority shall never cross the Provider Boundary.
